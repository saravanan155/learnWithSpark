## What this is

A **Week 3 "Mastering Agentic AI"** project (The Gen Academy). *Learn with Spark* is an AI-literacy
game for kids and visual-learning beginners. The learner receives **Spark**, a robot with an "empty
brain," and teaches it across levels — learning how AI actually works along the way. The engineering
substance: **the game levels are generated by a multi-agent LangGraph pipeline.** On an admin-only
page the owner types a concept; the pipeline researches it, drafts a kid-friendly lesson, generates a
playable **React** game for it, tests + repairs that code, and — after **three human approvals** —
publishes it to a database the learner side reads from. Published levels can be **revisited/edited
later (with versioning)**.

Guiding principle: **"write actions deserve a human."** Reads are autonomous; anything that
creates/publishes is gated.

## What this is NOT (prevents drift)

- ❌ Not a Scrum/Jira/master-data/SDLC tool.
- ❌ Not a tutor that narrates its own internals ("watch me use RAG"). The agentic engine is
  **hidden**; the learner just plays games.
- ❌ Lesson 3 is **knowledge cutoff**, NOT hallucination.
- ❌ The coding agent does not invent UI freely — it picks from a **fixed set of 4 mechanics**.

## Two surfaces

- **Learner side (keep simple):** loads published levels from the DB, plays lessons, tracks
  progress/mastery. First build just needs to load + play ONE generated level end-to-end.
- **Admin side (the agentic showpiece, owner-only):** owner types a concept; the pipeline co-creates
  a level *with* the human through three approval gates; publish; revise later.

## Requirements to demonstrate (course rubric)

- Build track: **LangChain + LangGraph (Python)**. The orchestration must be **our own** graph.
- **At least one model call via Nebius Token Factory.**
- Deliverables: **GitHub repo** (+ architecture diagram marking read vs write tools and the human
  gates), **demo video ≤ 5 min**, **documentation**.

| Concept | "Proving it" looks like |
|---|---|
| Multi-agent | 2+ specialized agents, each one job |
| Control flow | the graph decides what step comes next |
| State | data held + passed across steps |
| Tool usage | agents call tools; mark **read** vs **write** |
| Checkpoint / persistence | state survives steps *and a restart* |
| Human-in-the-loop | pauses for approval on write actions |
| Error / tool-failure recovery | a tool fails → retry / ask / stop, not crash |

Handout warning: *"If your agent works on the happy path but falls over on the first tool failure,
you have not finished."* → error handling + HITL are where the grade is.

## The three lessons (curriculum — finalized)

Spark starts with an empty brain. Arc: **See → Talk → Knows-its-limits.** Keep each lesson trivially
simple — a tiny game that flows flawlessly through the pipeline beats a complex one that breaks.

1. **"Teach Your Robot to See"** (AI learns from examples). Kid teaches Spark ~5 items, quizzes it,
   then shows it something it was never taught → "I don't know that — you never taught me!" Mechanic:
   drag & drop. **This is the hand-built WORKED EXAMPLE (see Build plan).**
2. **"Teach Your Robot to Talk"** (what an LLM is). Spark fails to finish "The cat sat on the ___"
   until the kid installs a glowing **"Language Brain"** module. Mechanic: fill-in-the-blank + upgrade beat.
3. **"Your Robot's Brain Has a Cutoff Date"** (knowledge cutoff, use **2023**). Sort cards into
   **"My Robot Knows"** (pre-2023/timeless) vs **"My Robot Doesn't Know"** (post-2023/real-time/personal).
   Mechanic: sort-into-two-columns (a drag & drop variant).

## Game mechanics (the fixed set the coding agent picks from)

ONE per level. This constraint is load-bearing — it keeps generation reliable and testable.

1. **drag_drop** (includes the sort-into-columns variant) · 2. **multiple_choice** · 3. **match_line**
(draw a line to match) · 4. **odd_one_out**.

## Tech stack (target — finalized)

| Layer | Choice |
|---|---|
| Agent orchestration | **LangGraph (Python)** — our own graph |
| Backend API | **FastAPI** — serves the graph; streams events over **SSE**; `/start` `/resume` `/stream` |
| Generated games | **React + Tailwind + Framer Motion** (plain HTML/JS is a *fallback only*) |
| Sandbox for generated code | **Sandpack** (`@codesandbox/sandpack-react`) — isolated iframe; the safety boundary (never `eval`) |
| Coding model | **Claude** (official `anthropic` SDK) — code generation + repair |
| Testing | **Deterministic code** (static + sandbox) **+ a DIFFERENT-family LLM judge** (Nebius, binary PASS/FAIL) |
| Cheap / required model | **Nebius Token Factory** — research, lesson text, guardrail, quality-judge (≥1 Nebius call is required) |
| State persistence | **LangGraph checkpointer** (`SqliteSaver` to start) |
| Database | **SQLite** to start — versioned published levels + learner progress |
| Tracing / evals | **LangSmith** |
| Tooling | **uv**, Python **3.12** |

## The component contract (MOST IMPORTANT ARTIFACT — build before any coding-agent code)

Every generated level is ONE self-contained React component matching this exact signature.

```tsx
interface GameLevelProps {
  onComplete: (result: { won: boolean; score: number }) => void;
  onProgress?: (step: string) => void;
}
export default function GameLevel({ onComplete }: GameLevelProps) { /* ... */ }
```

**Rails the coding agent MUST obey (verbatim in its prompt):**
- Export exactly ONE default component named `GameLevel` with that signature.
- Allowed imports ONLY: `react`, `framer-motion`, and the project's `<Spark mood="…" />` mascot.
  Tailwind via `className`. Nothing else.
- NO network/`fetch`, NO external asset URLs, NO `localStorage`/`sessionStorage`, NO `eval`.
- MUST call `onComplete({won, score})` when the game finishes; keyboard- AND touch-accessible.
- Fully self-contained: all state via React hooks; all data inline from the spec.
- Spark is rendered only via `<Spark mood="…" />` (never drawn by the level): show `spec.sparkMoods.start`
  on load, switch to `spec.sparkMoods.won` on success.

## Spark mascot (pre-made offline — the coding agent NEVER generates art)

6 fixed transparent PNGs in `/assets/spark/` (`curious, proud, confused, excited, unsure, confident`),
shown via a small `<Spark mood>` wrapper the owner builds. Mood map: L1 curious→proud, L2 confused→excited,
L3 unsure→confident. The agent only chooses *when* to show which mood.

## The admin agentic flow (three distinct human gates)

```
1. Owner types a concept (+ optional idea)
2. RESEARCH agent → kid-ready lesson (explain → game task → validate → feedback → recap); 2–3 options
3. [GATE 1 — PICK]            owner picks an idea (or re-research)
4. GUARDRAIL  (chosen idea only) → kid-safety + age + factual-accuracy
5. [GATE 2 — FINAL APPROVAL]  the misinformation checkpoint; approve to build (separate from Gate 1 on purpose)
6. CODING agent (Claude) → React GameLevel.tsx for the chosen mechanic
7. TESTING agent → static checks + sandbox tests (+ optional judge); REPAIR loop on failure (max 3) → escalate
8. [GATE 3 — PLAY-TEST]       owner plays the built level in Sandpack; approve/reject
9. PUBLISH → live in the DB
10. DB stores versioned levels + learner progress
```

**Why three gates:** idea-level, content-accuracy-level (catch misinformation *before* it becomes a
kids' game), and built-product-level. This layered review is a core selling point.

## LangGraph architecture (nodes + state)

```
research_node (Nebius)
  → [interrupt GATE 1 pick]
  → guardrail_node (chosen idea only)
  → [interrupt GATE 2 final approval]
  → write_tests_node (failing tests first, TDD)
  → coding_node (Claude → GameLevel.tsx)
  → static_check_node (deterministic)
  → test_node (sandbox run + optional Nebius judge)
       ├ pass → [interrupt GATE 3 play-test] → publish_node → DB
       └ fail → repair_node (Claude + EXACT error) → static_check_node
                 (repair_count ≥ 3 → escalate_node → human)
```

```python
class LevelCreationState(TypedDict):
    concept: str
    research_summary: str
    idea_options: list[dict]
    chosen_idea: dict | None
    guardrail_result: dict | None
    lesson_spec: dict | None
    failing_tests: str | None
    generated_code: str | None
    test_results: dict | None
    repair_count: int
    error_log: list[str]
    approved_for_build: bool
    play_test_approved: bool
    base_level_id: str | None     # set when revising an existing level
    revisit_mode: str | None      # "full" | "light" | None
```

Primitives: `SqliteSaver` (one `thread_id` per run), `interrupt()` + `Command(resume=...)` at each
gate (keep side effects AFTER the interrupt; nodes idempotent), `RetryPolicy(max_attempts=3)` +
`repair_count` cap + `escalate_node`.

## LessonSpec schema (research outputs → coding consumes)

```ts
interface LessonSpec {
  id: string; version: number; status: "draft" | "live" | "archived";
  lessonNumber: number; title: string; concept: string;
  mechanic: "drag_drop" | "multiple_choice" | "match_line" | "odd_one_out";
  story: string; prompt: string;
  items: Array<{ id: string; label: string; imageHint?: string }>;  // emoji/visual cue, NO URLs
  solution: any; feedback: { correct: string; incorrect: string };
  sparkMoods: { start: SparkMood; won: SparkMood };
  summary: string; ageTier: "kid" | "teen" | "beginner";
}
```

When `coding_node` fires, give Claude ALL of: the approved `LessonSpec` + the component contract +
**the hand-built Lesson-1 worked example** + mechanic-specific notes. Output: one `GameLevel.tsx` string.

## Testing (deterministic + a different-family judge)

Use a **different model family to test than to generate** (avoids self-preference blind spots).
- **Mechanical checks — NOT an LLM:** parses? imports in allowlist? exports `GameLevel`? renders
  without throwing? known-correct solution triggers a win? wrong answer triggers feedback?
- **Subjective judgment — Nebius, binary PASS/FAIL:** winnable? teaches the concept? age-appropriate?
- **TDD:** write the failing tests FIRST (from the spec), then run them against the generated code.

## Repair loop (the most gradeable behavior — protect time for it)

On test failure, feed the **exact error + failing test** back to Claude ("Your code failed this test
with this error. Fix it."), retry while `repair_count < 3`, else **escalate to human**. The specific
error feedback is what makes it converge instead of flailing.

## Revisit / edit a published level (versioning)

- **Version, never overwrite.** Level records carry `version` + `status` (`live`/`draft`/`archived`).
- **Atomic swap:** a new version goes `live` only after it clears **Gate 3**; the old flips `archived`.
- **Full revisit** (re-enter from the concept) vs **light revisit** (keep spec, regenerate code).
- **Never bypass the gates** — Gate 3 (play-test) is mandatory before the swap.
- Implementation: re-enter the same LangGraph flow with the published level as the starting state.

## Single success metric

% of generated levels approved with ≤ 1 round of edits (e.g. "5 generated, 4 clean, 1 tweak = 80%").

## Cautions

- **LangGraph changes fast** — pin versions; verify `interrupt()`/`Command`/`RetryPolicy`/checkpointer
  names against current docs.
- **Nebius catalog changes** — verify the model id is live in the playground; use the `/v1/` base URL.
- **Sandpack is the safety boundary** — never `eval` generated code in the main app.
- **The worked example is non-negotiable** — generated-level quality tracks the hand-built Lesson 1.
- **LLM-as-judge is imperfect** — keep its rubric binary; deterministic tests are the real safety net.
- **Keep the first generated concept trivially simple** — make the orchestration the star on demo day.
- **MCP is optional and last** — version-sensitive LangGraph integration; add only after the core loop works.
